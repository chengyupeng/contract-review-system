# 合同审查配置中心技术设计

**版本**: v3.0  
**日期**: 2026-06-22

---

# 一、概要

## 1.1 背景

合同审查系统需要灵活的审查规则管理能力。配置中心提供可视化工作台，支持：审查点（含维度、风险点判断标准）的维护，以及合同类型-立场与审查点的关联配置。AI 引擎在审查合同时按合同类型和立场直接查询 DB 加载对应审查点规则。

## 1.2 功能目标

**Tab 1 - 审查点维护**

- 维度管理：新增/编辑/删除维度，删除时校验是否有审查点引用
- 审查点 CRUD：按维度分组展示，支持关键词搜索
- 审查点详情：名称、维度、描述、审查说明、风险点列表（高/低/无风险判断标准）、审查示例、默认结论

**Tab 2 - 合同类型-立场配置**

- 合同类型 CRUD：编码、名称、立场、描述、启用状态
- 审查点关联：为合同类型-立场绑定审查点（按维度分组多选）
- 确认保存：全量替换该合同类型-立场的审查点关联

## 1.3 非功能目标

- 接口响应 P95 < 500ms，直接读 DB，无需缓存层
- 配置变更立即生效

## 1.4 方案要点

- 单库 MySQL，无缓存、无消息队列
- 四张表：维度 + 审查点 + 合同类型 + 关联表
- 风险点列表、审查示例、默认结论以 JSON 列存入审查点表（无独立查询场景，避免过度拆表）

# 二、整体设计

## 2.1 系统架构

```
┌─────────────────────────────────────┐
│           前端 SPA（配置中心）         │
│  Tab1: 审查点维护                     │
│  Tab2: 合同类型-立场配置              │
└──────────────────┬──────────────────┘
                   │ REST API / JSON
┌──────────────────▼──────────────────┐
│              后端服务                 │
│  DimensionService                    │
│  AuditPointService                   │
│  ContractTypeService                 │
└──────────────────┬──────────────────┘
                   │
┌──────────────────▼──────────────────┐
│               MySQL                  │
│  dimension / audit_point             │
│  contract_type / contract_type_audit_point │
└─────────────────────────────────────┘
                   ▲
                   │ 直接查询（按合同类型+立场）
┌──────────────────┴──────────────────┐
│              AI 审查引擎              │
└─────────────────────────────────────┘
```

## 2.2 模块职责

| 模块 | 职责 |
|------|------|
| DimensionService | 维度 CRUD，删除前校验引用 |
| AuditPointService | 审查点 CRUD，JSON 字段整体读写 |
| ContractTypeService | 合同类型 CRUD + 审查点关联全量替换（事务） |

## 2.3 数据流

**配置写入流**：管理员操作 → REST API → Service 层校验 → MySQL 持久化  
**AI 读取流**：AI 引擎传入合同类型+立场 → 查 `contract_type` 得到 id → 查关联表得到 `audit_point_id` 列表 → 批量读取审查点详情（含 JSON 字段）

# 三、数据库设计

## 3.1 ER 图

```
dimension
  id (PK)
  name
  description
  sort_order
    │
    │ 1 : N
    ▼
audit_point
  id (PK)
  dim_id (FK → dimension.id)
  name / description / instruction
  risk_points (JSON)
  examples (JSON)
  default_result (JSON)
  deleted_at
    │
    │ N : M (通过关联表)
    ▼
contract_type_audit_point
  contract_type_id (FK)
  audit_point_id (FK)
    ▲
    │ 1 : N
    │
contract_type
  id (PK)
  code / name / stance / description
  enabled / deleted_at
```

## 3.2 维度表 `dimension`

```sql
CREATE TABLE `dimension` (
  `id`          BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  `name`        VARCHAR(100)    NOT NULL COMMENT '维度名称',
  `description` TEXT                     COMMENT '维度说明',
  `sort_order`  INT             NOT NULL DEFAULT 0,
  `created_at`  DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at`  DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_name` (`name`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='审查维度';
```

## 3.3 审查点表 `audit_point`

```sql
CREATE TABLE `audit_point` (
  `id`             BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  `dim_id`         BIGINT UNSIGNED NOT NULL                  COMMENT '所属维度 ID',
  `name`           VARCHAR(200)    NOT NULL                  COMMENT '审查点名称',
  `description`    VARCHAR(500)                              COMMENT '描述，如：采购合同-甲方立场',
  `instruction`    TEXT                                      COMMENT '审查说明',
  `risk_points`    JSON                                      COMMENT '风险点列表',
  `examples`       JSON                                      COMMENT '审查示例列表',
  `default_result` JSON                                      COMMENT '默认结论 {level, analysis, suggestion}',
  `sort_order`     INT             NOT NULL DEFAULT 0,
  `created_at`     DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at`     DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  `deleted_at`     DATETIME                 DEFAULT NULL,
  PRIMARY KEY (`id`),
  KEY `idx_dim` (`dim_id`, `deleted_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='审查点';
```

**JSON 字段结构**：

`risk_points`：
```json
[
  {
    "name": "主体信息缺失",
    "highStd": "任一方名称、地址、证件号缺失",
    "lowStd": "存在但表述不规范",
    "noneStd": "双方主体信息完整且清晰"
  }
]
```

`examples`：
```json
[
  {
    "original": "甲方：（盖章处）",
    "level": "高风险",
    "analysis": "仅留盖章处，未填写具体主体信息",
    "suggestion": "补充完整的公司名称、法定地址"
  }
]
```

`default_result`：
```json
{
  "level": "高风险",
  "analysis": "合同主体信息不完整，存在较大法律风险",
  "suggestion": "要求对方补充完整主体信息"
}
```

## 3.4 合同类型表 `contract_type`

```sql
CREATE TABLE `contract_type` (
  `id`          BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  `name`        VARCHAR(100)    NOT NULL COMMENT '合同类型名称',
  `stance`      VARCHAR(50)     NOT NULL COMMENT '立场，如甲方/乙方',
  `description` TEXT                     COMMENT '描述',
  `enabled`     TINYINT         NOT NULL DEFAULT 1 COMMENT '1启用 0停用',
  `created_at`  DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at`  DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  `deleted_at`  DATETIME                 DEFAULT NULL,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_name_stance` (`name`, `stance`),
  KEY `idx_enabled` (`enabled`, `deleted_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='合同类型';
```

## 3.5 合同类型-审查点关联表 `contract_type_audit_point`

```sql
CREATE TABLE `contract_type_audit_point` (
  `id`               BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  `contract_type_id` BIGINT UNSIGNED NOT NULL,
  `audit_point_id`   BIGINT UNSIGNED NOT NULL,
  `sort_order`       INT             NOT NULL DEFAULT 0,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_ct_ap` (`contract_type_id`, `audit_point_id`),
  KEY `idx_ct` (`contract_type_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='合同类型-审查点关联';
```

# 四、API 接口设计

**基础路径**：`/api/v1/config`  
**通用响应格式**：`{ "code": 0, "msg": "ok", "data": ... }`

## 4.1 通用错误码

| code | HTTP 状态码 | 说明 |
|------|------------|------|
| 0 | 200 | 成功 |
| 40001 | 400 | 参数校验失败 |
| 40401 | 404 | 资源不存在 |
| 40901 | 409 | 名称/编码重复，或有引用无法删除 |
| 50001 | 500 | 内部服务错误 |

---

## 4.2 维度接口

### GET /dimensions — 获取维度列表

**响应**：

| 字段 | 类型 | 说明 |
|------|------|------|
| data | array | 维度列表 |
| data[].id | number | 维度 ID |
| data[].name | string | 维度名称 |
| data[].description | string | 维度说明 |
| data[].sortOrder | number | 排序 |

```json
{
  "code": 0,
  "data": [
    { "id": 1, "name": "基础检查", "description": "合同基本要素核查", "sortOrder": 0 },
    { "id": 2, "name": "交易条款", "description": "价格、付款、交付等条款", "sortOrder": 1 }
  ]
}
```

---

### POST /dimensions — 创建维度

**请求**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | string | 是 | 维度名称，唯一，≤100字 |
| description | string | 否 | 维度说明 |
| sortOrder | number | 否 | 排序，默认 0 |

```json
{ "name": "基础检查", "description": "合同基本要素核查", "sortOrder": 0 }
```

**响应**：

```json
{ "code": 0, "data": { "id": 1 } }
```

**错误**：名称重复 → `{ "code": 40901, "msg": "维度名称已存在" }`

---

### PUT /dimensions/:id — 更新维度

**请求**（同创建，字段均为可选）：

```json
{ "name": "基础核查", "description": "更新说明" }
```

**响应**：`{ "code": 0 }`

---

### DELETE /dimensions/:id — 删除维度

**业务规则**：存在关联审查点则拒绝删除。

**响应**：
- 成功：`{ "code": 0 }`
- 有引用：`{ "code": 40901, "msg": "该维度下存在审查点，无法删除" }`

# 五、关键业务逻辑

## 5.1 维度删除保护

```
DELETE /dimensions/:id
  → SELECT COUNT(*) FROM audit_point WHERE dim_id = ? AND deleted_at IS NULL
  → count > 0: 返回 40901
  → count = 0: DELETE FROM dimension WHERE id = ?
```

## 5.2 合同类型-审查点关联全量替换

配置审查点时采用全量替换策略，前端传来当前选中的完整列表，后端事务内先清后写，保证幂等。

## 5.3 AI 引擎读取链路

```sql
-- 1. 查合同类型
SELECT id FROM contract_type 
WHERE name = ? AND stance = ? AND enabled = 1 AND deleted_at IS NULL;

-- 2. 查关联审查点
SELECT audit_point_id FROM contract_type_audit_point 
WHERE contract_type_id = ? ORDER BY sort_order;

-- 3. 批量加载审查点详情（含 JSON 字段）
SELECT id, name, instruction, risk_points, examples, default_result 
FROM audit_point 
WHERE id IN (...) AND deleted_at IS NULL;
```
